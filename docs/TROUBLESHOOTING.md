# 개발 중 발생한 이슈와 해결 기록

방문자 로깅(구글시트 + Apps Script) 기능을 구축하면서 실제로 겪은 문제와 해결 과정을 시간 순으로 정리한다.

## 1. curl로 POST 테스트 시 `HTTP 405` + 구글 드라이브 오류 페이지

**증상**: Apps Script 웹앱에 `curl -X POST`로 테스트하면 302 리다이렉트를 따라간 뒤 "현재 파일을 열 수 없습니다"(구글 드라이브 접근 오류)와 함께 `HTTP 405`가 반환됨. 반면 같은 URL에 `GET`을 보내면 정상적으로 "doGet 함수를 찾을 수 없습니다"라는 Apps Script 자체 오류가 떴다 (이는 우리가 `doGet`을 안 만들었으니 정상 응답).

**원인 추정**: GET이 정상 도달한 것으로 보아 배포/접근 권한 자체는 문제가 없었다. curl이 Google의 POST 리다이렉트(302 → `script.googleusercontent.com/macros/echo`)를 처리하는 방식과, 실제 브라우저의 `sendBeacon`/`fetch(no-cors)`가 리다이렉트를 따라가는 방식이 달라서 생긴 curl 고유의 문제로 추정된다. 실제 브라우저의 `no-cors` 요청은 302를 자동으로 따라가 최종 목적지까지 완료하지만, curl은 `--post302`를 써도 같은 방식으로 동작하지 않았다.

**해결**: curl 검증을 포기하고, 실제 배포된 사이트에서 직접 클릭해보고 구글시트에 행이 쌓이는지 확인하는 방식으로 전환. 결과적으로 실제 브라우저에서는 정상 동작함을 확인했다.

## 2. Apps Script 구문 오류 (`SyntaxError: Unexpected token ')'`, 15번째 줄)

**증상**: `apps-script/Code.gs` 최신 내용을 편집기에 붙여넣은 뒤 저장하니 15번째 줄에서 구문 오류 발생. 소스 코드 자체는 괄호가 정확히 맞는 상태였다.

**원인 추정**: 기존 `Code.gs` 내용을 완전히 지우지 않고 이어붙였거나, 편집기의 자동 괄호 닫기 기능이 붙여넣기 과정에서 괄호를 중복 삽입한 것으로 추정.

**해결**: 전체 선택 후 삭제하고 새로 붙여넣기를 권장했고, 최종적으로는 사용자가 새 배포를 생성해 해결됨(배포 URL이 `AKfycbwyxEu9Du...` → `AKfycbz-uySEPLvuGm...`로 변경).

## 3. 새 배포로 인한 URL 재동기화

**증상**: 구문 오류를 새 배포로 우회하면서 웹앱 URL이 통째로 바뀜.

**원인**: "새 배포"를 만들면 새 URL이 발급된다. 같은 URL을 유지하려면 "배포 관리 → 기존 배포 수정 → 버전: 새 버전"으로 가야 하는데, 그 경로 대신 새 배포를 만든 것으로 보인다.

**해결**: `js/config.js`의 `APPS_SCRIPT_URL`을 새 URL로 갱신하고 커밋/푸시. 이후 `README.md`에 "코드 변경 시 배포 관리 → 새 버전 경로를 사용해 URL을 유지하라"는 절차를 명시해뒀다.

## 4. Apps Script URL/토큰이 배포된 사이트에서 노출되는 문제

**증상**: `js/config.js`의 `APPS_SCRIPT_URL`을 git에 커밋해도 되는지에 대한 우려 제기.

**분석**: 정적 사이트 특성상 브라우저 JS가 이 URL을 직접 호출해야 하므로, **git에 커밋하는지 여부와 무관하게 배포된 사이트의 페이지 소스/개발자도구 Network 탭에서 누구나 볼 수 있다.** gitignore로 숨겨도 배포 즉시 노출되는 정보라 근본적인 해결책이 아니다. 진짜로 완전히 숨기려면 서버(예: Vercel Serverless Function)를 두고 그 서버만 토큰을 알게 하는 구조가 필요하지만, GitHub Pages는 정적 파일만 서빙하므로 구조적으로 불가능하다.

**해결**: URL을 숨기는 대신, `doPost`에 공유 비밀 토큰(`INGEST_TOKEN`) 검증을 추가해 "URL을 우연히 발견한 봇/무작위 요청이 시트에 쓰레기 데이터를 넣는 것"만 방지하기로 결정 (강력한 보안은 아님, 의도적인 트레이드오프). 사용자는 이 수준의 보호로 충분하다고 판단해 GitHub Pages(정적 호스팅)를 그대로 선택했다.

## 5. 방문자 개인정보가 담긴 CSV 파일이 프로젝트 폴더에 있었음

**증상**: `git add -A`로 스테이징하던 중 `portfolio-visitor-log - Visitors.csv` 파일이 발견됨 — 실제 구글시트에서 내려받은 방문자 로그 CSV(36행)로, 20개 이상의 행에 실제 이름/이메일이 포함되어 있었다.

**원인**: 사용자가 구글시트를 CSV로 내보내 프로젝트 폴더에 저장해둔 것으로 보이며, Claude가 만든 파일이 아니었다. `git add -A`를 쓰면 이런 파일도 무심코 커밋될 위험이 있다.

**해결**: 커밋 전에 파일 내용을 확인해 PII 포함 여부를 검증했고, `.gitignore`에 `*.csv`를 추가해 앞으로 실수로 커밋되지 않도록 막았다. 파일 자체는 로컬에 그대로 남겨둠 (삭제하지 않음).

## 6. `clickCoordinates` 컬럼이 비어있음

**증상**: 24번째 컬럼(`clickCoordinates`)을 추가한 뒤 프로젝트 카드를 클릭해도 특정 행(3번 행)에서 좌표 값이 비어있었다. `projectClickSequence`(13번 컬럼)에는 값이 있는데 `clickCoordinates`만 비어있어 코드 버그처럼 보였다.

**분석**: `logProjectClick(projectId, x, y)` 함수는 `clickSequence`와 `clickCoordinates`를 항상 같은 함수 호출 안에서 함께 채우므로, 하나만 비고 하나만 차는 것은 코드 구조상 불가능했다. 클라이언트(GitHub Pages)와 서버(Apps Script `COLUMNS` 배열) 코드를 직접 확인한 결과 둘 다 최신 상태로 정확히 배포되어 있었다.

**진짜 원인**: 문제의 행은 문법 오류/구버전 배포가 살아있던 시점에 이미 만들어진 오래된 행이었다 — 그 시점의 `Code.gs`에는 `clickCoordinates`가 `COLUMNS`에 없었을 가능성이 높다.

**해결**: 새로고침 후 새로 카드를 클릭해 방금 생긴 최신 행을 확인하니 `clickCoordinates`에 좌표가 정상적으로 찍혔다. **교훈: Apps Script를 여러 번 재배포하는 동안 발생한 시트의 오래된 행과, 최신 코드로 만든 새 행을 혼동하지 않도록 항상 "가장 최근" 행을 기준으로 검증해야 한다.**

## 7. 누적 스냅샷 방식으로 인한 과대집계 위험

**증상**: 클릭/필터/스크롤/마우스 데이터가 세션 시작부터 계속 누적되어, 클릭하거나 페이지를 이탈할 때마다 "그 순간까지의 전체 히스토리"가 매번 통째로 새 행에 다시 기록되고 있었다. 방문자 로그를 분석한 `docs/visitor-log-analysis.md`에서도 "여러 행을 그대로 합산하면 과대집계된다"는 주의사항이 별도로 발견됨.

**문제점**: 예를 들어 사용자가 필터를 5번 클릭하면, 그 세션의 각 행마다 그때까지 클릭된 필터 배열이 점점 길어지며 반복 기록된다. 여러 행의 값을 단순히 더해서 "총 클릭 수"를 구하면 실제보다 훨씬 부풀려진 값이 나온다.

**해결**: `js/tracker.js`의 `flushNow()`가 전송 후 `filterClicks`, `clickSequence`, `clickCoordinates`, `dwellMap`, `mouseGrid`, `formStepStatus`를 모두 초기화하도록 변경. `pageDurationMs`도 "페이지 진입 후 총 시간"에서 "직전 전송 이후 경과 시간"으로 의미를 바꿨다. 이제 **한 행 = 그 시점에 새로 발생한 이벤트 1개**만 담으므로, 여러 행을 합산해도 정확하다. 단, `formCompleted`/`consentGiven`/리드 정보(이름·이메일 등)는 한 번 정해지면 계속 유지되는 상태값이라 예외적으로 매 행에 반복 포함시켰다 (개수를 세는 값이 아니라 과대집계 문제가 없기 때문).

이와 함께 `js/form.js`도 폼 단계 전환마다(진입/완료/이탈) 즉시 flush하도록 맞춰서, 클릭 이벤트와 동일하게 "이벤트 1개 = 행 1개" 원칙을 폼 흐름에도 일관되게 적용했다.
